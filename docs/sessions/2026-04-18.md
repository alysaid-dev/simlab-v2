## Implementasi

### P0 — Form submit wiring
- **SuratBebasLab**: handleSubmit → `POST /api/clearances` + redirect ke Riwayat
- **Transaksi.tsx** 5 handler wire:
  - `handleProses` → `PATCH /api/loans/:id/status = ACTIVE` (aktivasi loan APPROVED existing; deviasi dari spec "POST" karena UI secara eksplisit berkata "data mahasiswa yang sudah approved" — aktivasi, bukan create)
  - `handlePraktikumSubmit` → `POST /api/loans` type=PRACTICUM, status=ACTIVE (walk-in)
  - `handleExtensionSubmit` / `handleNewPerpanjangSubmit` → `PATCH /api/loans/:id` (body endDate)
  - `handleReturnSubmit` → `PATCH /api/loans/:id/status = RETURNED`
  - `handleMarkComplete` → `PATCH /api/loans/:id/status = RETURNED`
- **TransaksiHabisPakai**: sudah POST `/api/consumables/:id/transactions` dari session sebelumnya, verified
- **PeminjamanLaptop**: handleSubmit → `POST /api/loans` (assetId dari kode laptop, lecturerId dari dropdown)

### P0 — Notifikasi loan lifecycle
| Event | Penerima | Template |
|---|---|---|
| Loan dibuat | Mahasiswa + Dosen | `notifyLoanCreatedToMahasiswa` + `notifyLoanApprovalToDosen` |
| Status → `APPROVED_BY_DOSEN` | Mahasiswa | `notifyLoanApprovedByDosenToMahasiswa` |
| Status → `APPROVED` | Mahasiswa + semua Laboran aktif | `notifyLoanApprovedByKalabToMahasiswa` + `notifyLoanApprovedToLaboran` |
| Status → `REJECTED` dari PENDING | Mahasiswa | `notifyLoanApprovedByDosenToMahasiswa` (approved=false) |
| Status → `REJECTED` dari APPROVED_BY_DOSEN | Mahasiswa | `notifyLoanApprovedByKalabToMahasiswa` (approved=false) |

Template `LoanApprovedByDosen/Kalab` di-extend dengan param `approved?: boolean`
(default true). Saat false, subject dan copy berubah ke variant penolakan +
field `alasan?` opsional.

### P1 — Data integrity
- `loans.service.updateStatus` wrap di `prisma.$transaction`:
  - `ACTIVE` → Asset.status `BORROWED`
  - `RETURNED` / `REJECTED` / `CANCELLED` → Asset.status `AVAILABLE` (hanya kalau masih BORROWED, tidak override DAMAGED/MAINTENANCE)
  - Create dengan status=ACTIVE juga lock asset ke BORROWED (walk-in praktikum)
- `equipmentLoans.service`:
  - `create` validasi stok tiap item + decrement dalam transaksi
  - `updateStatus` transisi keluar "holding" (RETURNED/REJECTED/CANCELLED) → increment stok kembali
- `clearances.service.create` dipanggil oleh controller yang pre-check `usersService.getObligations` → 409 kalau ada tanggungan aktif

### Schema change

## Infrastruktur

### Systemd unit `simlab-v2-be.service`
Di-update jadi prod-correct (backup saya simpan di `/tmp/simlab-v2-be.service`
saat implementasi; user apply via sudo):

```ini
ExecStartPre=/usr/bin/npx --no-install prisma generate
ExecStartPre=/usr/bin/npm run build
ExecStart=/usr/bin/node dist/index.js
KillMode=mixed
TimeoutStopSec=15
```

Runtime kini compiled (`node dist/index.js`) bukan `tsx runtime`. Tiap
`git pull` → `sudo systemctl restart simlab-v2-be` → auto-build.

### Deploy workflow
```
# Backend update:
git pull
sudo systemctl restart simlab-v2-be
# Auto-run: prisma generate → tsc → run dist

# Frontend update:
npm run build
# Serve otomatis oleh nginx dari /var/www/simlab-v2/dist
# User hard refresh (Ctrl+Shift+R) untuk bypass cache

# Schema change:
cd backend
npx prisma db push
sudo systemctl restart simlab-v2-be
```

### Observability
```
systemctl status simlab-v2-be --no-pager
journalctl -u simlab-v2-be -f              # follow mode
journalctl -u simlab-v2-be -n 100 --no-pager
```
