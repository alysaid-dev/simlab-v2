## 3) Native `alert()` / `confirm()` → shadcn AlertDialog ✅

Semua native browser pop-up diganti modal dalam-app, konsisten styling dengan sisa app.

**Infra baru (`src/lib/dialog.tsx`):**
- `DialogProvider` (wrap di `App.tsx`) + `useDialog()` hook imperative:
  - `alert(message, opts?)` → `Promise<void>`
  - `confirm(message, opts?)` → `Promise<boolean>`
- Opsi: `title`, `confirmText`, `cancelText`, `destructive` (tombol merah).
- State dispatch single source → satu `<AlertDialog>` di root.

**Converted ~55 call sites di 16 file:**
- Pages: `Akun`, `Aset`, `MonitorTransaksi`, `PeminjamanAlat`, `PeminjamanLaptop`, `PeminjamanRuangan`, `PengaturanAplikasi`, `PersetujuanDosen`, `PersetujuanKepalaLab`, `PersetujuanLaboran`, `SuratBebasLab`, `Transaksi`, `TransaksiHabisPakai`.
- Components: `RiwayatDendaPanel`, `TerimaBarangPanel`, `DaftarBarangPanel`.

**Destructive variant** diterapkan untuk: Batalkan peminjaman (Monitor Transaksi), Tolak approval (PersetujuanKepalaLab/Laboran), Nonaktifkan akun (Akun), Hapus aset (Aset), Hapus consumable (DaftarBarangPanel), Hapus hari libur (PengaturanAplikasi).

Pattern call site: `const { alert, confirm } = useDialog();` di top component fn → shadow native globals. Existing `alert("msg")` tetap work (fire-and-forget), `confirm(...)` perlu `await` untuk boolean.

## 6) Bugfix — Kepala Lab approve Surat Bebas Lab error P2002 ✅

**Gejala:** "Gagal Memproses Data Sudah Ada (unique constraint)" saat kalab klik Setujui.

**Root cause:** `clearances.service.ts approveByKepalaLab` hitung `nomor_surat` dari `count(APPROVED within year) + 1`. Logic ini rapuh karena:
1. **Gap nomor** — DB punya nomor 002, 003, 004 (nomor 001 missing dari sesi awal).
2. **Legacy data** — 1 letter status `APPROVED` tapi `signed_at_kepala_lab` NULL (tidak ter-count karena filter tanggal) → count mismatch.
3. Count = 3 → assign 004 → collide dengan yang sudah ada.

**Fix:** ganti logic `count + 1` → `MAX(urut existing tahun ini) + 1`. Filter pakai `nomorSurat: { endsWith: '/YEAR' }` + parse prefix 3 digit pertama. Tahan gap + legacy data + idempotent.

File: `backend/src/services/clearances.service.ts`.

## Catatan teknis

- **AlertDialog imperative** — pola `useDialog()` hook + `DialogProvider` wrapping `<AuthProvider>` children di `App.tsx`. Setiap komponen yang butuh dialog: `const { alert, confirm } = useDialog();`. Native global `window.alert` / `window.confirm` TIDAK digunakan lagi di seluruh app.
- **Equipment loan walk-in** — laboran boleh pass `userUid` di body POST `/api/equipment-loans` untuk assign loan ke peminjam lain. Session identity tetap harus LABORAN+ (guard di controller). Status default masih `PENDING`, tapi walk-in kirim `status: "ACTIVE"` supaya langsung lewat tanpa approval chain.
- **Notif fire-and-forget** — semua `notify*` di controller di-`catch()` ke `console.error` supaya gagal kirim notif tidak gagalkan request utama. Pattern sama dengan laptop loan existing.
- **`/api/me/active-items`** — single endpoint aggregate 5 source dengan `Promise.all`. Hindari N+1 round trip; mahasiswa panggil sekali pas Dashboard load. Return `counts` supaya FE bisa conditional render tanpa re-compute.
- **Bucket MAX vs count** di clearance — prinsip umum: untuk sequential numbering, pakai MAX existing instead of count. Tahan gap, tahan data legacy tanpa nomor, tahan manual edit DB.
