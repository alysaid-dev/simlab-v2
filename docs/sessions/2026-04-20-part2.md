# SIMLAB V2 — Progress 20 April 2026 (Part 2)

Sesi lanjutan: **debugging 403 Kepala Lab** yang ternyata bukan RBAC melainkan
Cloudflare WAF memblokir method `PATCH`. Solusi: method-override via helper
frontend + middleware backend.

## Root cause

Kepala Lab login sukses, dashboard load, role terdeteksi benar — tapi PATCH
`/api/clearances/:id/status` selalu 403. Log `journalctl -u simlab-v2-be`
**tidak mencatat request sama sekali**, dan response header menunjukkan
`server: cloudflare`, `content-type: text/html`, `cf-ray: 9ef1ee079f6d4f85-JOG`.

Artinya: Cloudflare WAF meng-intercept PATCH ke `statistics.uii.ac.id/simlab/api/**`
sebelum sampai Node.js. Awalnya dicurigai gara-gara VPN (IP reputation), tapi
user tes tanpa VPN juga 403 — jadi bukan IP, melainkan method PATCH atau path
pattern yang kena rule WAF.

IT BSI UII biasanya menjawab "konfigurasi ada di VPS masing-masing" untuk kasus
seperti ini (historis: SSO, connectivity). Jadi fix dikerjakan dari sisi aplikasi.

## Solusi: method-override

Frontend kirim PATCH sebagai **POST + header `X-HTTP-Method-Override: PATCH`**;
middleware backend rewrite `req.method` kembali ke PATCH sebelum router cocokkan.
Cloudflare cuma lihat POST di kabel → lolos. Semantik REST di backend tetap PATCH.

### Perubahan backend

`backend/src/index.ts`:

- CORS `allowedHeaders` tambah `"X-HTTP-Method-Override"`.
- Middleware baru (sebelum `morgan` dan `app.use("/api", ...)`):

  ```ts
  app.use((req, _res, next) => {
    if (req.method === "POST") {
      const override = req.header("x-http-method-override")?.toUpperCase();
      if (override === "PATCH") req.method = "PATCH";
    }
    next();
  });
  ```

Route handler `router.patch(...)` tidak diubah. Zero churn di 10 route.

### Perubahan frontend

File baru `src/lib/apiFetch.ts` — drop-in pengganti `fetch`:

```ts
export async function apiFetch(input, init) {
  const method = init?.method?.toUpperCase();
  if (method !== "PATCH") return fetch(input, init);
  const headers = new Headers(init?.headers);
  headers.set("X-HTTP-Method-Override", "PATCH");
  return fetch(input, { ...init, method: "POST", headers });
}
```

Lalu di 8 file yang punya `method: "PATCH"`, import `apiFetch` + replace
`fetch(` → `apiFetch(` (replace_all, karena apiFetch drop-in compatible
untuk method non-PATCH).

File terdampak:
- `src/components/WaNumberOnboardModal.tsx`
- `src/pages/Transaksi.tsx`
- `src/pages/PersetujuanKepalaLab.tsx`
- `src/pages/PersetujuanLaboran.tsx`
- `src/pages/PersetujuanDosen.tsx`
- `src/pages/MonitorTransaksi.tsx`
- `src/pages/Akun.tsx`
- `src/pages/HabisPakai.tsx`

`AuthContext.tsx` sengaja tidak diubah (tidak ada PATCH di situ).

## Verifikasi

Log setelah deploy — Kepala Lab pakai VPN:


Semua PATCH 200, middleware override bekerja end-to-end, termasuk alur approve
yang trigger generate PDF + hash Kepala Lab.

## Catatan operasional

- Workaround ini bisa dipakai permanen, tidak ada downside teknis.
- Kalau suatu hari WAF rule diperbaiki, tinggal hapus isi `apiFetch.ts` (atau
  kembalikan ke PATCH asli) — frontend tetap kompatibel karena semua call site
  sudah pakai helper.
- Kalau ada endpoint PATCH baru di masa depan, **pakai `apiFetch` di frontend**
  atau WAF akan block lagi. Middleware backend sudah otomatis menangani
  semua PATCH.

## Pattern untuk debug 403 mendatang

Kalau user lapor 403 di UI:
1. **Cek `journalctl -u simlab-v2-be`** — kalau PATCH tidak tercatat = Cloudflare.
2. **Cek response headers di DevTools** — `server: cloudflare` + `content-type: text/html` = Cloudflare block.
3. Baru setelah itu curigai RBAC.
