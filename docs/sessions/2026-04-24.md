## 1. Bug RBAC — akun DOSEN dapat card "Petunjuk Tendik"

### Observasi

User lapor: akun test yang di-centang **DOSEN** saja di `/akun` tetap
melihat card "Petunjuk Tendik" di dashboard (yang seharusnya STAFF-only).

API response `/api/auth/me`:

```json
{
  "uid": "241003314",
  "affiliation": ["Staff"],
  "memberOf": [],
  "roles": ["STAFF", "DOSEN"]
}
```

### Root cause

`backend/src/middleware/auth.ts` `deriveRoles()` ambil roles dari 4 sumber
lalu UNION-kan:

1. `memberOf` (LDAP groups) via `ROLE_MAP`
2. `affiliation` via `ROLE_MAP`
3. `dbRoles` dari `user_roles` table
4. MAHASISWA auto-email, SUPER_ADMIN uid-list

ROLE_MAP lama mapping `"staff" → STAFF`. UII Shibboleth IdP kasih
`affiliation=["Staff"]` ke **semua karyawan** (dosen/tendik/laboran sama)
— mapping generic itu over-grant. Admin yang centang DOSEN saja di UI
tetap dapat STAFF dari Shibboleth.

### Keputusan: Opsi A3

Tiga opsi dibahas:

| Opsi | Behavior | Risk |
|---|---|---|
| A1 strict | dbRoles only, ignore Shibboleth kecuali MAHASISWA email + SUPER_ADMIN uid | User baru 0 role, harus admin assign |
| A2 hybrid | dbRoles override kalau ada, fallback Shibboleth | Tetap over-grant saat first login |
| **A3** | Hybrid + hapus mapping generic (`staff/dosen/faculty` dst) dari ROLE_MAP | 1 user existing kena (Linda Agustina, baru 1 hari, belum di-assign) |

Pilih **A3** karena: konsisten (non-mahasiswa butuh admin approve),
backward-compatible untuk mahasiswa (auto-email tetap jalan), migrasi
risk cuma 1 user.

### Verifikasi DB sebelum deploy

```
total_users: 42
users_with_role_assignment: 41
users_without_role_assignment: 1  (Linda Agustina, non-student)
```

Mahasiswa existing aman karena `users.service.ts:46`
`upsertFromShibboleth()` auto-insert MAHASISWA role ke `user_roles` tiap
`/api/auth/me`. Jadi 14 mahasiswa udah semua di-record di DB, bukan cuma
derived runtime.

### Implementasi

`ROLE_MAP` di `auth.ts:48` dipangkas jadi cuma 6 marker `simlab-*`:

```ts
const ROLE_MAP: Record<string, Role> = {
  "simlab-super-admin": "SUPER_ADMIN",
  "simlab-kepala-lab": "KEPALA_LAB",
  "simlab-dosen": "DOSEN",
  "simlab-laboran": "LABORAN",
  "simlab-staff": "STAFF",
  "simlab-mahasiswa": "MAHASISWA",
};
```

Build + restart backend. Verify via `/api/auth/me` — sekarang return
`roles: ["DOSEN"]` saja untuk akun test. Card "Petunjuk Tendik" hilang
dari dashboard DOSEN.

### Komit

- `f080858` fix(rbac): dbRoles authoritative — hapus mapping generic
  Shibboleth affiliation

Push ke origin.

## 4. Notifikasi WhatsApp tidak sampai (tapi email sampai)

### Observasi

User tolak reservasi via super admin → email sampai (setelah cek spam),
WA nggak sampai. Log backend nggak ada jejak error — karena call-site
pakai `void notifyX()` (fire-and-forget), exception ter-swallow.

### Investigasi

1. **Cek DB**: reservasi tercatat REJECTED dengan alasan benar. Backend
   PATCH `/api/reservations/.../status` return 200. Notif fire tapi
   silent fail.
2. **Cek env**: `WA_API_URL` + `WA_API_TOKEN` ada, Fonnte provider.
3. **Curl Fonnte langsung**:
   - Attempt 1 (4 endpoint sekaligus): 3 dari 4 return **502 Bad
     Gateway**
   - Attempt 2: validate=200, send=200, success message id 153288795
     masuk queue, delivered ke HP user
4. **Kesimpulan**: Fonnte API flaky — intermittent 502 transient.

### Fix

Dua perubahan di `backend/src/services/notification/`:

**a. Logging di `dispatch` (`index.ts:90`)**

Setiap notif log:

```
[notif] to=EMAIL/PHONE subj="..." email:STATUS wa:STATUS
```

STATUS = `sent` / `skipped(reason)` / `failed(reason)`. Level `error`
kalau ada yang failed, else `log`. Sekali ubah, benefit ke semua 13+
helper notif.

**b. Retry di `sendWhatsApp` (`transports.ts`)**

- Max 3 attempts, backoff `[0, 1000, 3000]` ms antar attempt
- Retry hanya untuk 5xx response atau network error (no response)
- 4xx (request salah / auth expired / nomor invalid) langsung throw —
  percuma retry
- Total worst-case ~4 detik sebelum give up

Tiap retry log:

```
[fonnte] retry 2/3 for 081215481452 after 1000ms
```

### Komit

- `58bcc06` fix(notif): WA retry-on-5xx + log hasil dispatch per
  call-site
