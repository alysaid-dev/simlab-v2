# 2026-04-23 part 3 ŌĆö Notifikasi pengajuan ruangan (full wiring)

## Konteks

User melaporkan: notifikasi pengajuan peminjaman ruangan tidak muncul
ke siapa pun (pemohon, laboran, kepala lab). Investigasi menemukan
3 masalah berurutan:

1. Helper `notifyNewReservationToLaboran()` ada di
   `reservations.controller.ts:153` tapi **tidak pernah dipanggil** dari
   `create()` handler.
2. Meski dipanggil, recipient hardcoded `"-"` (placeholder lama) ŌĆö pasti
   ditolak nodemailer.
3. Tidak ada notif ke pemohon (mahasiswa) di flow manapun. Pola loans
   sudah punya `notifyLoanCreatedToMahasiswa` + `notifyLoanApprovalToDosen`
   + notif tiap state transition, tapi reservasi belum ada padanannya.

Schema sebenarnya cukup ŌĆö `Room.laboratoryId` + `Laboratory.kepalaLab` +
`LaboratoryLaboran` sudah ada, jadi recipient bisa di-resolve runtime
tanpa migration.

## Keputusan operasional (dialog dengan user)

| Trigger | Kirim ke |
|---|---|
| `create` (PENDING) | LABORAN lab pemilik room + pemohon (konfirmasi) |
| `CHECKED` (laboran setuju) | KEPALA_LAB lab tsb + pemohon (info menunggu kalab) |
| `APPROVED` (kalab setuju) | pemohon + LABORAN lab tsb (siapkan ruangan & jadwal) |
| `REJECTED` (laboran/kalab tolak) | pemohon saja, dengan alasan |

User minta: laboran di tahap APPROVED = **semua** laboran lab itu (bukan
hanya yang verifikasi di tahap CHECKED) ŌĆö alasan: kalau yang verifikasi
sedang cuti, lab tetap punya orang yang siap. REJECTED tidak perlu
kabar laboran.

## Implementasi

### 1. Template notif baru (templates.ts)

Tambah 4 template di akhir file:

- `roomReservationCreatedToMahasiswa` ŌĆö konfirmasi pemohon
- `roomReservationCheckedToMahasiswa` ŌĆö info pemohon "menunggu kalab"
- `roomReservationDecisionToMahasiswa` ŌĆö disetujui/ditolak (varian
  approved=true/false, copy berbeda + opsional `alasan`)
- `roomReservationApprovedToLaboran` ŌĆö kalab approved, laboran siapkan
  ruangan

Pola HTML/WA mengikuti template existing (emailLayout, detailTable,
WA_SALAM, waDetails). Subject prefix `[SIMLAB]`.

### 2. Dispatch helper (notification/index.ts)

Tambah 4 helper:
- `notifyRoomReservationCreatedToMahasiswa`
- `notifyRoomReservationCheckedToMahasiswa`
- `notifyRoomReservationDecisionToMahasiswa`
- `notifyRoomReservationApprovedToLaboran`

### 3. Service (reservations.service.ts)

- `listInclude` di-extend: `user` tambah `waNumber`, `room` tambah
  `laboratoryId`, tambah `laboratory.kepalaLab` (id, name, email,
  waNumber) + `laboratory.laborans.user` (id, name, email, waNumber,
  isActive). Dipakai resolver recipient di controller.
- `create()` sekarang fetch `Room.laboratoryId` lalu populate
  `RoomReservation.laboratoryId` saat insert. Sebelumnya field itu null
  di semua row reservasi.
- `updateStatus()` terima `rejectionReason?: string`. Kalau status
  REJECTED, persist ke kolom `rejectionReason` (kolom sudah ada di
  schema sejak awal, tapi belum pernah ditulis).

### 4. Controller (reservations.controller.ts)

- Helper `notifyNewReservationToLaboran` lama dihapus (placeholder
  `"-"`).
- Tambah util lokal `formatTanggalPinjam` (Intl id-ID weekday + tanggal)
  & `formatWaktuPinjam` (HH:mm ŌĆō HH:mm WIB, timeZone Asia/Jakarta).
- `statusBody` zod sekarang menerima `rejectionReason: z.string().trim()
  .min(1).max(500).optional()`. Sebelumnya field ini silently dropped
  meski frontend kemungkinan akan kirim.
- `create` handler: setelah service.create, kirim notif ke pemohon +
  loop laboran aktif di lab tsb (filter `isActive && (email||waNumber)`).
- `updateStatus` handler: branching per ReservationStatus:
  - CHECKED ŌåÆ notif pemohon (CheckedToMahasiswa) + kalab
    (RoomReservationToKalab dengan recipient asli, bukan `"-"` lagi)
  - APPROVED ŌåÆ notif pemohon (DecisionToMahasiswa approved=true) + loop
    semua laboran aktif (ApprovedToLaboran)
  - REJECTED ŌåÆ notif pemohon saja (DecisionToMahasiswa approved=false +
    alasan)

Semua pakai pola `void notifyXxx(...)` fire-and-forget seperti loans.
