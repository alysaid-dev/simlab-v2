## 2. Insiden Cloudflare: upload PDF 403 dari non-VPN

### Laporan

Laboran coba submit pengajuan Peminjaman Ruangan → muncul error 403
dengan HTML response "Attention Required! | Cloudflare" + Ray ID
`9f0b0d78fec84f85`. Request tidak pernah sampai backend (log
`journalctl -u simlab-v2-be` kosong untuk waktu kejadian).

### Diagnosa bertahap

1. **Awal**: saya duga content-based multipart block (mirip kasus
   `cloudflare_patch_block.md` kemarin). Usulkan workaround JSON+base64
   kalau tidak bisa konfigurasi CF.
2. **Update user**: "saya bisa mengunggah melalui VPN" → saya pivot ke
   hipotesis **IP-based** (CF block IP ISP rumahan, trust VPN UII).
3. **Koreksi dari SOC UII**: setelah user kirim email ke SOC dengan Ray
   ID, balasan SOC menyebutkan file pertama "dicurigai mengandung
   script berbahaya mungkin html atau xss". User diarahkan coba PDF
   lain di jaringan seluler biasa.

**Diagnosa yang benar**: dua faktor gabungan —
1. CF Managed Rule (XSS Detection) scan body multipart, match pattern
   `<script`, `javascript:`, `onload=`, dsb. PDF pertama yang
   di-upload berisi pattern tersebut (mungkin embed JS macro, form
   field dengan action, atau komentar mirip script).
2. CF apply tier berbeda per network: VPN UII = trusted, Managed Rules
   relaxed/skipped. Jaringan publik = full inspection. Inilah sebabnya
   file sama lolos via VPN tapi kena via seluler.

Total 11× block (angka dari dashboard CF yang di-share SOC) — semuanya
konsisten ke rule XSS detection.

### Artefak: email permohonan ke SOC

Draft email dikirim dari user (Muhammad Aly Sa'id) ke Security
Operation Center BSI UII. Format: versi yang "tidak paham keamanan CF"
— posisinya pengguna yang melaporkan kendala dan minta arahan, bukan
network engineer yang prescribe solusi. Berisi: URL endpoint, method,
Ray ID, fakta "via VPN lolos", catatan singkat bahwa endpoint di
belakang Shibboleth + validasi Zod + multer PDF-only + size cap 200KB.

### Langkah berikutnya (in progress saat catatan ini)

- User sedang test PDF bersih di jaringan seluler
- Kalau lolos → content-specific, edukasi user soal PDF tanpa JS embed
- Kalau tetap gagal → eskalasi lagi (minta SOC Skip WAF Managed Rules
  khusus path `/simlab/api/reservations` POST)

### Memory update

## 3. Bug tampilan: tanggal sidang "-" di persetujuan surat bebas lab

### Laporan

Laboran melihat detail mahasiswa pengajuan surat bebas lab → field
"Tanggal Sidang" tampil `"-"` sementara Nama dan NIM muncul benar.
User konfirmasi tanggal sidang wajib diisi mahasiswa saat submit.

### Akar masalah

- Backend `/api/clearances` **sudah mengembalikan** field `tanggalSidang`
  (Prisma `findMany` tanpa `select` = semua kolom)
- Frontend di `PersetujuanLaboran.tsx:294` dan `PersetujuanKepalaLab.tsx:378`
  **hardcode** `tanggalSidang: "-"` dengan komen lama `"tidak tersimpan
  di model"` yang sudah tidak valid
- Tipe `BackendClearance` tidak mendeklarasikan field

### Fix

- `src/pages/PersetujuanLaboran.tsx` (baris 41-49 + 294):
  - Tambah `tanggalSidang: string | null` ke interface
  - Ganti hardcode dengan `c.tanggalSidang ? formatDateFull(c.tanggalSidang) : "-"`
- `src/pages/PersetujuanKepalaLab.tsx` (baris 265-271 + 378):
  - Fix yang sama

### Catatan: backend validation masih opsional

`clearances.controller.ts:28-31` zod masih `tanggalSidang: z.string().optional()`.
Frontend form menandai required (`<span className="text-red-500">*</span>`),
tapi backend tidak enforce. Belum di-tighten — menunggu user test apakah
fix display ini sudah cukup, atau perlu strengthen validation juga.
