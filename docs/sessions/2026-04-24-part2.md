## 2. Bug: URL gambar petunjuk kehilangan prefix `/simlab`

User tes upload gambar via Manajemen Petunjuk, berhasil dapat markdown
`![...](/api/guides/image/gambar-...png)`. User lapor bahwa link relatif
tidak render di browser, hanya link full-URL yang muncul.

### Root cause

- `backend/src/controllers/guides.controller.ts:209` return
  `url: "/api/guides/image/<filename>"` — **tanpa** prefix `/simlab`.
- Frontend `src/pages/ManajemenPetunjuk.tsx:506` paste URL itu apa adanya
  ke markdown: `![alt](/api/guides/image/<filename>)`.
- Browser resolve path itu dari origin root →
  `https://statistics.uii.ac.id/api/guides/image/...` → **404**.
- URL yang benar: `https://statistics.uii.ac.id/simlab/api/guides/image/...`
  (dibuktikan via curl: `/api/...` = 404, `/simlab/api/...` = 200).

### Kenapa upload sendiri tetap sukses

Upload pakai `fetch(\`${API_BASE}/api/guides/image\`, ...)` di mana
`API_BASE = VITE_API_URL = "/simlab"` dari `.env`. POST-nya ke
`/simlab/api/guides/image` → oke. Tapi `<img src>` di markdown TIDAK
lewat `API_BASE`.

### Status

**Belum di-fix**. Diusulkan dua patch ke user (belum ditugaskan — user
pilih tidak perlu gambar dulu untuk petunjuk yang sedang disusun):

**A. Forward fix** di `ManajemenPetunjuk.tsx:506`:
```ts
insertAtCursor(`\n\n![${altText}](${API_BASE}${data.url})\n\n`);
```

**B. Backward-compat** di `PetunjukReader.tsx` — custom `img` component di
ReactMarkdown yang auto-prefix `/api/...`:
```tsx
<ReactMarkdown
  components={{
    img: ({ src, ...rest }) => (
      <img src={src?.startsWith("/api/") ? `${API_BASE}${src}` : src} {...rest} />
    ),
  }}
>
```

Opsi B penting kalau ada konten petunjuk yang sudah terlanjur disimpan
dengan URL relatif salah — biar tidak perlu edit manual.

Ini TODO untuk sesi selanjutnya kalau user akan mulai tempel gambar di
petunjuk beneran.

## 3. Bug Modul Peminjaman Alat (Aset) — tombol CRUD semua stub

User lapor: "tambah alat tidak muncul". Investigasi di
`src/pages/PeminjamanAlat.tsx` tab **Aset** menemukan **3 bug
bersamaan**:

| Aksi | Status sebelum | Lokasi |
|---|---|---|
| Tambah Alat | `console.log` + alert palsu, tidak hit backend | `:1442-1452` |
| Edit Alat | Sama — stub, tidak hit backend | `:1533-1542` |
| Hapus Alat | Button tanpa `onClick` sama sekali | `:1353-1355` |

Padahal backend sudah lengkap: `POST/PATCH/DELETE /api/equipment`
(`backend/src/routes/equipment.routes.ts`), `equipment.controller.ts`
dengan Zod body validator, service layer pakai Prisma.

### Fase 1 — wire CRUD ke backend (tanpa schema change)

Patch awal:

1. Tambah `confirm` ke `useDialog()` destructure.
2. Tambah `CONDITION_TO_BACKEND` map (label Indonesia → enum Prisma
   `AssetCondition`): Baik → GOOD, Rusak Ringan → MINOR_DAMAGE, Rusak
   Berat → MAJOR_DAMAGE. "Hilang" di-reject di frontend (tidak ada di
   backend enum).
3. Tambah 3 handler async: `handleAddTool`, `handleEditTool`,
   `handleDeleteTool`. Semua call `refetchEquipment()` setelah sukses,
   error dari backend ditampilkan lewat `alert()` dialog.
4. Delete pakai `confirm({ destructive: true })` — dialog merah.
5. Button disabled + teks "Menyimpan..." selama request.
6. Modal Tambah: hapus field "ID Alat" karena `Equipment.id` auto-gen
   UUID, user tidak bisa input.

### Fase 2 — user complaint: "Nama Alat terisi 12218, ID Alat UUID jelek"

User tes tambah alat, sukses tersimpan, tapi:
- Kolom "ID Alat" di tabel menampilkan UUID panjang
  (`579ab423-c8b2-4f4f-bf4f-67f33e91f0bd`)
- User memasukkan "12218" di field yang dilabeli "Nama Alat" karena
  tidak ada tempat lain untuk kode custom

Akar masalah struktural: **`model Equipment` tidak punya field `code`**.
Kontras dengan `model Asset` (laptop) yang punya `code` terpisah dari
`id`.

Tiga opsi dibahas:

| Opsi | Keputusan |
|---|---|
| A — tambah kolom `code String? @unique` | **Dipilih** |
| B — repurpose field `category` yang sudah ada | Ditolak (semantik salah, tidak unique) |
| C — tampilkan UUID dipotong | Ditolak (tetap tidak bisa kode custom) |

### Fase 3 — Option A: schema change

**Prisma (`backend/prisma/schema.prisma:247-264`)**: tambah
```
code String? @unique
```
di atas `name`. Nullable karena ada 1 row existing ("12218") yang belum
punya kode — akan di-set manual via UI Edit.

**`npx prisma db push`** warning "data loss" karena unique constraint
baru — false alarm untuk nullable unique. Pakai `--accept-data-loss`
dan sukses dalam 331ms. Prisma Client regenerated otomatis. Fakta ini
disimpan ke memory `prisma_db_push_nullable_unique.md` karena cukup
sering ketemu.

**Backend** (`equipment.controller.ts:14`): tambah
`code: z.string().trim().min(1).optional()` ke `createBody`. `updateBody`
auto-dapat via `.partial()`. Service layer tidak perlu diubah — pakai
`Prisma.EquipmentCreateInput` generic.

**Frontend** (`src/pages/PeminjamanAlat.tsx`):
- `BackendEquipment` interface: tambah `code: string | null`
- `Tool` interface: tambah `code: string` (empty string untuk legacy
  row yang null di DB)
- Dua mapping `data.items.map(e => ...)` (initial + refetch): tambah
  `code: e.code ?? ""`
- `toolForm` state + reset-on-open: tambah `code: ""`
- Set saat klik edit pensil: tambah `code: tool.code`
- Modal **Tambah**: field baru "Kode Alat" (required, placeholder
  `ALT-001`), di atas "Nama Alat"
- Modal **Edit**: field readonly "ID Alat" (UUID) DIGANTI jadi editable
  "Kode Alat" — supaya user bisa isi kode untuk legacy row
- Tabel: kolom "ID Alat" tampilkan `tool.code || <span italic
  gray>(belum ada)</span>` — fallback untuk legacy
- `handleAddTool` + `handleEditTool`: validasi kode tidak kosong, kirim
  ke body POST/PATCH

Typecheck frontend + backend clean. Build frontend:
- `index-D5Ci5cJD.js` (buggy) → `index-BikF0R5r.js` (fixed)
- Ukuran: 974 kB uncompressed / 254 kB gzip

Backend restart: `sudo -n systemctl restart simlab-v2-be` → aktif

`ExecStartPre`.
