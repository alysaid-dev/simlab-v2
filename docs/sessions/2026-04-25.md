## 1. B1 — URL gambar petunjuk hilang prefix `/simlab`

Akar masalah: backend `guides.controller.ts:209` return
`url: "/api/guides/image/<filename>"` (tanpa `/simlab`). Markdown yang
disimpan jadi `![alt](/api/...)` → browser resolve dari origin root →
404 di `https://statistics.uii.ac.id/api/...` (yang benar
`/simlab/api/...`).

Diskusi dua pendekatan dari sesi 24 April:
- **A. Forward fix** di `ManajemenPetunjuk.tsx:506` — prefix `${API_BASE}` saat insert markdown
- **B. Defense-in-depth** di renderer — img component override yang prefix saat render

Dipilih **B saja**. Alasan:
- Menyimpan URL `/api/...` di DB lebih portable (deployment-agnostic) — kalau mount path ganti dari `/simlab/` ke yang lain, cukup ubah env, tidak perlu migrate konten
- Kalau A saja: stored content ber-`/simlab/...` jadi leak deploy detail ke DB

Implementasi: tambah `components.img` ke 2 ReactMarkdown call site:
- `src/components/PetunjukReader.tsx:140-159` — reader (mahasiswa/dosen lihat petunjuk)
- `src/pages/ManajemenPetunjuk.tsx:667-686` — preview di editor

Logic identik:
```tsx
img: ({ src, ...rest }) => (
  <img
    src={typeof src === "string" && src.startsWith("/api/")
      ? `${API_BASE}${src}`
      : src}
    {...rest}
  />
)
```

Tidak di-extract jadi helper karena cuma 2 call site (rule "three similar
lines is better than premature abstraction"). Kalau nanti muncul renderer
ke-3, baru pertimbangkan ekstraksi.

## 4. Pagination server-side di Akun.tsx (page size 25)

User observasi data Akun cukup banyak — perlu pagination. Diskusi
beban server: server-side pagination justru **lebih ringan** dari status
quo karena per-response payload mengecil (25 vs 200+ row).

**Silent bug** yang tersingkap saat audit: `Akun.tsx:218` fetch
`/api/users` **tanpa parameter `take`** → backend default `take=50`
(`users.service.ts:90`). Artinya kalau lab punya >50 user, **50 user
pertama doang yang tampil** di tab Akun, sisanya tak terlihat. Sudah
silently broken.

### Backend changes

`users.controller.ts:8-17` — `listQuery` Zod schema tambah:
```ts
isActive: z
  .enum(["true", "false"])
  .transform((v) => v === "true")
  .optional(),
```

(Pakai enum + transform daripada `z.coerce.boolean()` karena coerce
nge-anggap any non-empty string sebagai true.)

`users.service.ts:82-104` — params type tambah `isActive?: boolean`,
`where` clause tambah `...(isActive !== undefined ? { isActive } : {})`.

### Frontend changes

`src/pages/Akun.tsx`:

1. Import `Pagination`, `PaginationContent`, `PaginationItem`,
   `PaginationLink`, `PaginationPrevious`, `PaginationNext`,
   `PaginationEllipsis` dari shadcn `ui/pagination`
2. Helper top-level `buildPageList(current, total)`:
   - ≤7 halaman → tampilkan semua
   - >7 → windowed `1 ... current-1 current current+1 ... total` dengan ellipsis
3. State baru: `currentPage`, `total`, `debouncedSearch`, konstanta `PAGE_SIZE = 25`
4. 3 useEffect:
   - Debounce search 300ms (`setSearchQuery` per-keystroke → `setDebouncedSearch` setelah 300ms idle)
   - Reset `currentPage = 1` saat `debouncedSearch` atau `statusFilter` berubah (mencegah out-of-range)
   - Fetch saat `currentPage`/`debouncedSearch`/`statusFilter` berubah
5. `fetchAccounts` baru: bangun query string dengan `URLSearchParams` (skip, take, search?, isActive?)
6. Hapus `getFilteredAccounts` (filter sudah di server). Render `accounts.map(...)` langsung.
7. JSX baru di bawah Card tabel: indikator "Menampilkan 1–25 dari 150 akun" + `<Pagination>` shadcn dengan Prev/page numbers/Next, disabled state via `pointer-events-none opacity-50`

Worst case 2 fetches saat user pindah filter dari halaman >1 (filter
change → fetch dengan old page+new filter, lalu setCurrentPage(1) →
fetch lagi). Acceptable — dua query kecil <100ms apart, kedua hasil
nimpa pertama. Tidak worth complexity untuk hindari.

Edge case yang dibiarkan: hapus akun terakhir di halaman terakhir →
hasil refetch kosong dengan `total > 0`. User klik Prev manual. Bisa
di-fix dengan auto-decrement, tapi rentan infinite loop kalau salah
deps. Skip.
