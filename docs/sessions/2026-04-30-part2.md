## 1. Bug — Input NIM tab Pengajuan tidak respons Enter

User uji modul Transaksi Laptop tab Pengajuan: ketik NIM lalu tekan
Enter → tidak ada respons. Harus klik tombol Cari supaya search jalan.

### Trace

`src/pages/Transaksi.tsx:730-737` — Input `id="nim"`:
- `value={searchNIM}` ✅
- `onChange={(e) => setSearchNIM(e.target.value)}` ✅
- `onKeyDown` ❌ tidak ada
- Tombol Cari (line 740-756) `onClick={handleSearch}` + `disabled={!searchNIM || searchState === "loading"}`

Bandingkan tab "Cari Peminjam" di file yang sama (`Transaksi.tsx:1612-1621`)
— sudah punya pola yang benar:

```tsx
onKeyDown={(e) => {
  if (e.key === "Enter") void handleLookupPeminjam();
}}
```

Jadi pola Enter sudah ada di file ini, cuma tab Pengajuan miss.

### Fix

`src/pages/Transaksi.tsx:730-741` — tambah `onKeyDown` dengan guard
yang mirror disabled-prop tombol:

```tsx
onKeyDown={(e) => {
  if (e.key === "Enter" && searchNIM && searchState !== "loading") {
    void handleSearch();
  }
}}
```

Pertimbangan guard: Cari Peminjam (line 1615) tidak guard, tapi
`handleLookupPeminjam` juga tidak. Untuk konsistensi UX dengan tombol
Cari yang punya `disabled={!searchNIM || searchState === "loading"}`,
saya replikasi guard yang sama di Enter handler — supaya spam Enter
saat empty/loading tidak double-fire.

### Verifikasi

`npx tsc --noEmit` clean. `npm run build` sukses, bundle hash baru
`index-zhCl5ckj.js`. Frontend-only, tidak perlu restart BE.

## 2. Audit pagination lintas-halaman

Carry-over dari TODO sesi 25 April (#7). User minta audit halaman
mana yang belum punya pagination, supaya bisa prioritaskan implementasi.
Delegasi ke Explore subagent supaya cepat.

helper `buildPageList(current, total)` + fetch dengan `URLSearchParams
{ skip, take, ... }` dari backend yang return `{ items, total }`.
PAGE_SIZE = 25.

Halaman yang fetch tanpa `take` param → silent capped at 50.

### Real risk assessment

- **PersetujuanKepalaLab Laptop tab** — saat musim TA, antrian
  approval dosen-approved laptop loans bisa cepat >50. Konsekuensi:
  kalab tidak melihat sebagian permohonan, mahasiswa stuck.
- **PersetujuanLaboran Reservasi tab** — batch booking ruangan beruntun
  (mis. seminggu sebelum UAS) bisa >50.
- **SuratBebasLab Riwayat** — super admin lihat akumulasi historis,
  cepat lewat 50.
- **PeminjamanRuangan** — user lihat reservasi mereka sendiri,
  biasanya kecil per user. Tapi role super admin lihat semua → high.

### Rekomendasi pendekatan (saya pitch ke user)

Per filosofi codebase ("3 similar lines = abstraction OK"), 7 halaman
P0+P1 dengan struktur mirip (tab + filter status + pagination) worth
ekstrak `usePaginatedList<T>` hook + komponen `<DataTablePagination>`
wrapper — tidak perlu copy-paste 5×.

Saran urutan:
1. **PersetujuanKepalaLab dulu** (highest impact: dataset paling cepat tumbuh) — pakai copy-paste pattern Akun
2. Setelah #1 kelar, refactor jadi hook (saat pola sudah stabil)
3. Apply hook ke 4 P0 sisanya
4. P1 di sesi/PR berpisah

Status sesi: user belum konfirmasi mau mulai dari mana. Audit selesai,
implementasi ditunda.
